# 堆利用和栈溢出的区别与联系

1. **栈溢出 vs. 堆利用——常见程度**  
   - **栈溢出**：CTF中经典、常见，但真实生产环境因金丝雀、ASLR、栈不可执行等防护机制，直接利用变难。仍多见于老旧系统、IoT、嵌入式设备。  
   - **堆利用**：在现代主流系统（Linux/Windows）中**更常见**，因为堆管理复杂，易出漏洞（UAF、堆溢出、双重释放等）。

2. **哪个更“致命”？**  
   - 两者一旦成功都可导致远程代码执行（RCE），破坏力相当。  
   - 但堆漏洞在现代目标上**出现频率更高、更难完全防御**，因此常被认为更具实战威胁。

3. **结论**  
   - **“堆利用在实战中更常见”** → ✅ 正确（针对现代高防护环境）。  
   - **“堆利用更致命”** → 不完全正确，致命程度相当；堆因出现多、利用难、防御难，所以更令防御者头疼。

4. **补充**  
   - 真实 pwn 领域还有其他常见漏洞：条件竞争、格式化字符串、整数溢出（常作为前置利用步骤）。


## 栈溢出
---

### 一、基础利用（无防护或防护不全的环境）

| 类型 | 原理 | 利用目标 | 典型场景 |
|------|------|----------|----------|
| **直接覆盖返回地址** | 覆盖栈上保存的返回地址，跳转到shellcode或后门函数 | 立即执行恶意代码 | 无NX、无ASLR、无金丝雀的老系统或CTF入门题 |
| **覆盖变量** | 覆盖栈上的局部变量（如权限标志、函数指针） | 改变程序逻辑（如提权、绕过认证） | 程序有判断变量但无边界检查 |

---

### 二、绕过栈金丝雀（Canary）

| 方法 | 原理 | 条件 |
|------|------|------|
| **泄露Canary** | 利用格式化字符串、越界读等先泄露栈上的canary值，再在溢出时原样写回 | 存在信息泄露漏洞 |
| **逐字节爆破** | 在fork子进程中，逐字节猜测canary（因子进程canary相同） | 程序会fork且无其他随机化 |
| **劫持异常处理** | 覆盖栈上的异常处理结构（如SHE），在canary检查失败前触发异常 | Windows SEH或Linux的__stack_chk_fail可被劫持 |
| **覆盖canary的检查位置** | 修改canary检查函数指针或绕过检查逻辑 | 特定漏洞组合（较少见） |

---

### 三、绕过NX（栈不可执行）—— ROP链

当栈不可执行时，不能直接执行shellcode，需用**返回导向编程（ROP）**。

| 类型 | 原理 | 特点 |
|------|------|------|
| **ret2text** | 跳转到程序本身的代码段中的后门函数或gadget | 最简单，无需泄露 |
| **ret2shellcode** | 将shellcode放入可执行内存（如堆、数据段），再跳转 | 需要先找到可写可执行区域 |
| **ret2libc** | 跳转到libc中的函数（如system、execve） | 需泄露libc基址（绕过ASLR） |
| **ret2syscall** | 直接调用系统调用（int 0x80或syscall） | 适用于静态编译或gadget充足 |
| **ret2csu** | 利用__libc_csu_init中的gadget进行万能ROP | 用于调用任意函数 |
| **SROP** | 利用sigreturn机制伪造signal frame | 可一次性设置所有寄存器 |

---

### 四、绕过ASLR（地址随机化）

ASLR使代码段、libc、堆、栈的基址随机化，需先泄露地址。

| 方法 | 原理 |
|------|------|
| **泄露GOT表项** | 通过溢出或信息泄露，读取已解析的函数地址，计算libc基址 |
| **利用栈残留数据** | 栈上可能残留之前调用过的函数返回地址或libc地址 |
| **部分覆盖** | 只覆盖返回地址的低2字节或1字节，在随机化范围内爆破 |
| **vsyscall & vDSO** | 某些固定地址（如vsyscall）不受ASLR影响（已逐步淘汰） |

---

### 五、高级/组合利用

| 类型 | 说明 |
|------|------|
| **栈迁移** | 将栈指针（rsp/esp）迁移到可控区域（如堆、bss），解决栈空间不足或需要更大ROP链的问题 |
| **ROP + 栈迁移 + 堆喷** | 结合堆布局，适用于栈空间受限或防护严格的环境 |
| **one gadget** | 利用libc中的单条execve("/bin/sh")地址，通过设置特定条件直接跳转 |

---

### 六、特殊环境利用

| 环境 | 特点 | 常用手法 |
|------|------|----------|
| **嵌入式/IoT** | 防护弱或无 | 直接覆盖返回地址执行shellcode |
| **Windows** | SEH、VEH异常处理 | 覆盖SEH链、绕过GS（类似canary） |
| **内核栈溢出** | 权限更高但利用复杂 | 劫持返回地址或函数指针，提权 |

---

### 总结表（按防护机制分类）

| 防护 | 利用方法 |
|------|----------|
| **无防护** | 直接覆盖返回地址 → shellcode |
| **有Canary** | 泄露canary / 爆破 / 劫持异常处理 |
| **有NX** | ROP / ret2libc / ret2syscall |
| **有ASLR** | 泄露地址（GOT、栈残留） + 计算基址 |
| **有Canary+NX+ASLR** | 泄露canary + 泄露libc基址 + ROP链 |
| **全防护+PIE** | 还需泄露代码段基址，通常通过栈残留或部分覆盖爆破 |




## 堆利用
---

## 一、基础漏洞类型

| 漏洞类型        | 原理                            | 利用目标                  |
| ----------- | ----------------------------- | --------------------- |
| **堆溢出**     | 向堆块写入超过其大小的数据，覆盖相邻堆块的元数据或用户数据 | 修改相邻块指针、伪造堆块、实现任意地址读写 |
| **释放后使用**   | 释放堆块后仍使用其指针（悬垂指针）             | 通过重新分配占用原位置，控制被使用的数据  |
| **双重释放**    | 对同一堆块调用两次free，破坏空闲链表结构        | 可导致任意地址分配（通过伪造堆块）     |
| **内存耗尽/耗尽** | 耗尽堆空间或触发特殊行为                  | 结合其他漏洞利用              |

---

## 二、glibc ptmalloc 特有的利用手法

针对主流Linux环境（glibc堆管理器），按利用目标分类：

### 1. 任意地址分配（Write-What-Where）

| 手法                      | 原理                                         | 目标                         |
| ----------------------- | ------------------------------------------ | -------------------------- |
| **unlink 利用**           | 伪造空闲堆块的fd/bk指针，在unlink时向指定地址写入堆地址          | 修改全局指针、函数指针                |
| **fastbin dup**         | 利用双重释放或UAF，使fastbin链中出现同一块两次，再分配得到目标地址     | 向任意地址写入（需伪造fastbin chunk）  |
| **tcache dup**          | 类似fastbin dup，但tcache机制更简单，同一块可多次放入链       | 任意地址分配（tcache poisoning）   |
| **large bin attack**    | 利用large bin的插入机制，向任意地址写入堆地址                | 常用于绕过一些保护                  |
| **unsorted bin attack** | 利用unsorted bin的fd/bk指针，向任意地址写入main_arena地址 | 泄露libc地址或修改global_max_fast |

### 2. 信息泄露（读任意内存）

| 手法 | 原理 |
|------|------|
| **泄露libc基址** | 释放块进入unsorted bin后，其fd/bk指向main_arena，读出可计算libc基址 |
| **泄露堆基址** | 通过UAF或堆溢出读取相邻块的指针，泄露堆的起始地址 |
| **泄露代码段地址** | 如果堆上有函数指针或返回地址 |

### 3. 控制流劫持（RCE）

| 手法 | 原理 |
|------|------|
| **__free_hook / __malloc_hook** | 覆盖glibc的钩子函数指针（如__malloc_hook），在malloc/free时跳转到shellcode或one gadget |
| **虚表劫持** | C++对象、FILE结构体中的虚函数指针被覆盖 |
| **exit钩子** | 覆盖_IO_2_1_stdout_等FILE结构中的vtable |

---

## 三、针对不同bins的利用

| Bin类型 | 特点 | 利用手法 |
|---------|------|----------|
| **fastbin** | 单链表，LIFO，大小<0x80 | fastbin dup, fastbin poisoning |
| **tcache** | 单链表，每线程独立，无完整性检查 | tcache dup, tcache poisoning, tcache stash |
| **unsorted bin** | 双向循环链表，大小合并后放入 | unsorted bin attack, 泄露libc |
| **small bin** | 双向链表，大小固定 | small bin attack (较少见) |
| **large bin** | 双向链表，有fd_nextsize/bk_nextsize | large bin attack, 泄露堆地址 |

---

## 四、高级/组合利用

| 手法 | 说明 |
|------|------|
| **House of系列** | 经典堆利用技巧合集，如House of Spirit、House of Force、House of Einherjar等 |
| **House of Force** | 利用top chunk的size字段被篡改，实现任意地址分配 |
| **House of Spirit** | 伪造一块假的fastbin块，欺骗free将其加入链表 |
| **House of Einherjar** | 利用off-by-one触发块合并，实现任意地址写 |
| **House of Orange** | 在没有free的情况下制造unsorted bin块（通过修改top chunk） |
| **House of Pig** | 结合large bin attack和IO_FILE利用 |
| **IO_FILE利用** | 劫持_IO_FILE结构体的vtable，触发任意函数调用 |

---

## 五、针对不同版本的glibc

| glibc版本 | 特性 | 影响 |
|-----------|------|------|
| **≤2.23** | fastbin无检查、unlink无保护 | 经典unlink、fastbin dup直接可用 |
| **2.23~2.26** | 添加tcache（2.26+） | tcache成为主要攻击面 |
| **2.27~2.29** | tcache增加count字段 | tcache dup需处理count |
| **2.30+** | tcache key、safe linking | 增加了异或加密和完整性检查 |
| **2.32+** | safe linking（fd/bk异或加密） | 泄露堆地址成为必要步骤 |
| **2.34+** | 移除__malloc_hook/__free_hook | 需转向其他目标（如_IO_file_jumps） |

---

## 六、特殊环境/应用

| 环境 | 特点 | 常用手法 |
|------|------|----------|
| **Windows堆** | 与glibc不同，有LFH、Segment Heap | 类似原理，但具体结构不同 |
| **jemalloc** | FreeBSD、Android、Rust使用 | 利用其bins和扩展块特性 |
| **tcmalloc** | Chrome、gperftools使用 | 利用span和size class |
| **内核堆（SLUB/SLAB）** | 内核内存分配器 | UAF + 伪造对象 + 提权 |

---

## 总结表（按glibc版本+手法）

| glibc版本 | 常用手法 |
|-----------|----------|
| **2.23及以下** | unlink、fastbin dup、unsorted bin attack、House of系列 |
| **2.26-2.29** | tcache dup、tcache poisoning、tcache stash |
| **2.30-2.31** | safe linking绕过（需泄露堆地址）、House of pig |
| **2.32+** | 仍然可用，但需额外泄露堆地址来解码safe linking |
| **2.34+** | 放弃钩子，转向IO_FILE、exit机制、_dl_fini等 |
